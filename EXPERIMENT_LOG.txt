2. PIPELINE (src/qwen3_comp/)
================================================================

Single entry point: run_inference.py → scripts/build_submission.py:run_inference()

Default production settings (run_inference.py / run_validation.py):

  Prompts:
    primary: strict
    retry:   commit_now (only if no extractable boxed answer)

  Self-consistency:
    n_mcq = 5 samples per MCQ question
    n_free = 3 samples per free-form question
    majority vote on normalized boxed answers

  Retry:
    max_retries = 1
    retry uses larger max_new_tokens (budget.py: 6144–8192)
    retry n = max(1, n_mcq or n_free // 2)

  Sampling:
    temperature=0.7, top_p=0.95, top_k=20, seed=0

  vLLM engine (vllm_runtime.py):
    quantization=bitsandbytes, dtype=bfloat16
    max_model_len=16384, gpu_memory_utilization=0.75 (OOM-safe default)
    max_num_seqs=96, enforce_eager optional (--enforce_eager if sampler OOM)
    backend=vllm (fallback: transformers + 4-bit)
    model_id default: Qwen/Qwen3-4B-Thinking-2507 (--model_id for HF checkpoint)

  Token budgets (route_max_new_tokens, retry=0):
    MCQ short/medium/long: 2048 / 3072 / 4096
    Free-form: 2048–4096 by difficulty and [ANS] count
    Retry: 6144 or 8192

  Batching (2026-05-29 update):
    --batch_size N (default 16): pending rows processed in chunks of N
    Per batch: primary vLLM generate() → retry generate() for failures in batch
    Checkpoint: append to responses_jsonl + rewrite submission.csv after EACH batch
    Resume: skip any id already present in responses_jsonl (last line wins on dupes)
    Tradeoff: batch_size 16 ~13 min/batch but ~12h full run; 64 ~30–40% faster;
              943 (monolithic) ~2.5–3h but loses all in-flight work on pod death

  Cause:

  Fix (runtime, no git pull):
    export VLLM_WORKER_MULTIPROC_METHOD=spawn

6. TIMING LESSONS (why full run >> initial estimate)
================================================================

Initial estimate (~6–8 h):
  - Linear scale from mini (466s × 943/15)
  - Assumed low effective tok/s, small-batch overhead per question

Actual full-run behavior:
  - Single batched vLLM call for 943 prompts → ~900–950 aggregate output tok/s
  - Progress bar counts completions (3429 primary = samples not questions)
  - Engine init amortized once
  - Effective ~1.5–2 h primary + ~0.5–1 h retry

Rule of thumb after this project:
  - Small-n validation time does NOT linearly predict full-batch time
  - Trust vLLM progress bar ETA on 3429 completion scale for primary pass
  - batch_size 16: ~13 min/batch, ~59 batches → ~12–13 h total
  - batch_size 64: ~15 batches, est ~7–8 h remaining from scratch
  - batch_size 943: ~2.5–3.5 h but one pod death loses entire in-flight chunk
  - Retry pass per batch is slow (~7.5 min) even when primary is ~5 min
  - vLLM progress: fast primary bar then slow retry bar is EXPECTED behavior

9. QUALITY / PROMPT NOTES (from milestone + mini)
================================================================

- Main failure mode at low budget: truncation before \boxed{}
- strict prompt default helps boxed extraction vs baseline
- commit_now retry for missing boxed answers
- MCQ: vote on letter inside \boxed{}
- Free-form: comma-separated subanswers in one \boxed{} matching [ANS] order
- Mini 86.7% boxed < 95% script warning threshold — expect some empty/bad rows
- Full private run: 661/943 boxed (70.1%); 635/943 sane+voteable (~67% judger-ready)
- 282 no_boxed_answer: truncation mid-reasoning, rarely complete \boxed{}
- 17 empty_boxed_answer: \boxed literal in text but truncated before closing brace
- 9 wrong_subanswer_count: comma grouping wrong for multi-[ANS] (e.g. 41,35,16 x4 ids)
- Filter for rerun: use rec.get("boxed") truthy, NOT "is not None" (keeps empty strings)
- Numerical near-misses may score wrong (e.g. 442.86 vs 442.857...) — judger strictness

Future improvements (next iteration):
  - Add a second repair stage for stubborn no-box rows (n=2 repair samples)
  - Failure-specific retries (no_box vs wrong_subanswer_count) with custom budgets
  - Two-stage: solve then deterministic format-only extractor prompt
  - Optional: stricter free-form answer-count correction for multi-[ANS]
