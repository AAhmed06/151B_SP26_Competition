2. PIPELINE (src/qwen3_comp/)
================================================================

Default production settings (build_submission.py / run_validation.py):

  Prompts:
    primary: strict
    retry:   commit_now (only if no extractable boxed answer)

  Self-consistency:
    n_mcq = 5 samples per MCQ question
    n_free = 3 samples per free-form question
    majority vote on normalized boxed answers

  Retry:
    max_retries = 1
    retry uses larger max_new_tokens (budget.py: 6144–8192)
    retry n = max(1, n_mcq or n_free // 2)

  Sampling:
    temperature=0.7, top_p=0.95, top_k=20, seed=0

  vLLM engine (vllm_runtime.py):
    quantization=bitsandbytes, dtype=bfloat16
    max_model_len=16384, gpu_memory_utilization=0.85
    backend=vllm (fallback: transformers + 4-bit)

  Token budgets (route_max_new_tokens, retry=0):
    MCQ short/medium/long: 2048 / 3072 / 4096
    Free-form: 2048–4096 by difficulty and [ANS] count
    Retry: 6144 or 8192

  Batching:
    One vLLM generate() for ALL pending questions (primary pass)
    Second generate() for retry failures only
    responses.jsonl written AFTER both passes complete (not per-question streaming)

3. DSMLP ENVIRONMENT — ISSUES & FIXES
================================================================

Issue A: numpy 2.x vs conda scipy/sklearn (BROKEN)
  Symptoms:
    AttributeError: _ARRAY_API not found
    numpy.core.multiarray failed to import
    transformers 5.x pulled via ~/.local
  Cause:
    Old install_env.sh installed to ~/.local with numpy 2.2.6, transformers 5.9.0
    No project .venv
  Fix:
    bash dsmlp/bootstrap_venv.sh   (or install_env.sh after git pull)
    Always: source .venv/bin/activate before python
    Pin: numpy>=1.26.4,<2, transformers>=4.51,<5, scipy/sklearn in venv
  Verify:
    python -c "import numpy, transformers, torch; print(numpy.__version__, transformers.__version__)"
    Expect: 1.26.x  4.xx  2.6.x+cu124

Issue B: Disk quota exceeded during install
  Cause:
    bootstrap installed accelerate BEFORE vllm → torch 2.12 + CUDA 13 (~3GB)
    then vllm pulled torch 2.6 + CUDA 12 again
    ~/.local had ~5.8GB junk from prior --user installs
  Fix:
    rm -rf ~/.local/lib/python3.11/site-packages/{torch,nvidia,vllm,transformers}*
    python3 -m pip cache purge
    rm -rf .venv
    bootstrap_venv.sh now: numpy/scipy first, then single pip install -r requirements-dsmlp.txt
    export PIP_NO_CACHE_DIR=1 during install

Issue C: vLLM "Cannot re-initialize CUDA in forked subprocess"
  Cause:
    torch.cuda.is_available() in parent before vLLM forks engine
  Fix (code):
    multiprocessing.set_start_method("spawn") in smoke/build_submission/run_validation
    _require_cuda() uses torch.cuda.device_count() not is_available()
  Fix (runtime, no git pull):
    export VLLM_WORKER_MULTIPROC_METHOD=spawn

Issue D: Blackwell / MIG pods
  vLLM + PyTorch 2.6 may fail on sm_120 / MIG slices
  Fix: launch-scipy-ml.sh -g 1 -c 8 -m 32 -v a30

Issue E: git pull "Disk quota exceeded"
  Free space first (free_disk.sh), then pull

Issue F: tmux "duplicate session: submit"
  Session already exists; use tmux attach -t submit
  Job may run inside existing session even if current shell has empty $TMUX

6. TIMING LESSONS (why full run >> initial estimate)
================================================================

Initial estimate (~6–8 h):
  - Linear scale from mini (466s × 943/15)
  - Assumed low effective tok/s, small-batch overhead per question

Actual full-run behavior:
  - Single batched vLLM call for 943 prompts → ~900–950 aggregate output tok/s
  - Progress bar counts completions (3429 primary = samples not questions)
  - Engine init amortized once
  - Effective ~1.5–2 h primary + ~0.5–1 h retry

Rule of thumb after this project:
  - Small-n validation time does NOT linearly predict full-batch time
  - Trust vLLM progress bar ETA on 3429 completion scale for primary pass

9. QUALITY / PROMPT NOTES (from milestone + mini)
================================================================

- Main failure mode at low budget: truncation before \boxed{}
- strict prompt default helps boxed extraction vs baseline
- commit_now retry for missing boxed answers
- MCQ: vote on letter inside \boxed{}
- Free-form: comma-separated subanswers in one \boxed{} matching [ANS] order
- Mini 86.7% boxed < 95% script warning threshold — expect some empty/bad rows
- Numerical near-misses may score wrong (e.g. 442.86 vs 442.857...) — judger strictness
